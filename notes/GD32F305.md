## 1. MCU Communication — USART mapping

### U13 USARTs (initialized in FUN_08003fa8)

| USART | Base | Baudrate | GPIO AF | DMA | Connected to |
|-------|------|----------|---------|-----|-------------|
| USART0 | `0x40013800` | `0x38400` = 230400 | `0x25` | channel 4 | **U16** — JSON, OTA, commands |
| USART1 | `0x40004400` | `0x1c200` = 115200 | `0x26` | channel 5 | Unknown (3rd MCU?) |
| USART2 | `0x40004800` | `0x1c200` = 115200 | `0x34` | channel 2 | Unknown |

### Protocol U13 ↔ U16 (USART0 @ 230400)

OTA packet structure (verified via `FUN_08008cb8`):
```
[2 bytes: length LE] [N bytes: data] [1 byte: XOR checksum]
```
Checksum = XOR of all data bytes.

Used for:
- OTA firmware download (baud can switch to 460800/921600)
- Status forwarding (`FUN_08007c94` sends status to U16)
- Sensor data in JSON format (via cJSON library on U16)

### Communication chain

```
Display Board (ESP32)
    │ UART @115200, binary protocol (0xAA 0x55, XOR CS)
    │ cmd 0x01-0x04 = button presses
    │ cmd 0x0B = PWD_VERIFY (4-digit PIN)
    │ cmd 0x0C = PWD_RESULT (response from mainboard)
    │ cmd 0x0D-0x15 = status, mow, charge, bat info etc.
    ▼
U16 (GD32F303) — board MCU
    │ Translates binary commands → JSON messages
    │ UART @230400, JSON via cJSON
    ▼
U13 (GD32F305) — main MCU
    │ Parses JSON, executes commands (motor, sensors, PIN verify)
    │ Sends JSON responses back
```

## 2. PIN Code — Storage and Verification

The PIN is a **4-digit** number stored in the **KV-store cache in RAM**
(address `0x2000027C`) and persisted in **EEPROM U22**.

### PIN verification flow (reconstructed from U16.md + ESP32.md)

```
1. Display shows "0   " (blinking)
   → User sets digit with +/- buttons
   → "0 0  " → "0 0 0 " → "0 0 0 0"
   
2. ESP32 sends frame to U16:
   0xAA 0x55 0x0B "1234" <XOR_CS>
   (cmd 0x0B = PWD_VERIFY, payload = 4 ASCII chars)

3. U16 forwards to U13 as JSON:
   {"cmd":"verify_pin","pin":"1234"}

4. U13 reads stored PIN from KV cache (RAM `0x2000027C`)
   → If match: returns OK
   → If not: returns FAIL

5. U16 → ESP32: 0xAA 0x55 0x0C 0x00 <CS> (OK)
                       or 0xAA 0x55 0x0C 0x01 <CS> (FAIL)

6. ESP32 displays:
   "IdLE" (OK — unlocked)
   "LoCK" (FAIL — stays locked)
```

### ESP32 NVS keys

| NVS key | Value | Meaning |
|---------|-------|---------|
| `robot_password` | `88888888` | WiFi/MQTT password, **NOT the PIN** |
| `robot_ssid` | `cy-public` | WiFi network name |
| `wifi_passwd` | `88888888` | Same WiFi password |
| `robot_name` | `MyMower` | Device name |
| `robot_sn` | `2312CGF250600035167` | Serial number |
| `iot_mqtt_uri` | `mqtt://server.sk-robot.com` | MQTT broker |

ESP32 verifies the password for WiFi/MQTT (`Check password succeeded/failed`),
while the 4-digit PIN is sent to the mainboard for verification.

## 3. ESP32 Communication Protocol

ESP32 has a full command set beyond just button forwarding:

| ID | Direction | Name | Description |
|----|-----------|------|-------------|
| `0x01` | ESP→MB | `BTN_UP` | Button + |
| `0x02` | ESP→MB | `BTN_DOWN` | Button - |
| `0x04` | ESP→MB | `BTN_OK` | Button OK |
| `0x0B` | ESP→MB | `PWD_VERIFY` | Send PIN (4 ASCII bytes) |
| `0x0C` | MB→ESP | `PWD_RESULT` | Verification result (0=OK, 1=fail) |
| `0x0D` | ESP→MB | `STATUS_REQ` | Query status |
| `0x0E` | MB→ESP | `STATUS_RSP` | Status (mow/charge/idle/lock/error) |
| `0x0F` | ESP→MB | `MOW_START` | Start mowing |
| `0x10` | ESP→MB | `CHARGE_RET` | Return to station |
| `0x11` | ESP→MB | `DISPLAY_OFF` | Turn off display |
| `0x12` | MB→ESP | `ERROR_INFO` | Error code |
| `0x14` | ESP→MB | `BAT_INFO_REQ` | Query battery |
| `0x15` | MB→ESP | `BAT_INFO_RSP` | Battery voltage (mV) |

Additionally — if WiFi is configured — ESP32 connects to MQTT broker
`server.sk-robot.com` and relays cloud commands to the mainboard.

The HTTP server strings (`Mower Log` HTML page with JS) are part of
the ESP32 firmware, not U13.

## 4. USB Mass Storage — Pendrive Handling

### What actually works with a pendrive:

```
1. User inserts USB pendrive
2. FUN_08002f18 detects USB ("USB disk Ready")
3. FUN_08003cdc creates log_yyyymmdd_hhmmss.html ("save log to usb disk")
4. FUN_0800303c (IAP handler) searches for SNK_*.bin files:
   → "MB IAP start ..."  when SNK_MB_*.bin  — firmware Main Board
   → "BB IAP start ..."  when SNK_BB_*.bin  — firmware Boundary Board
   → "DB IAP start ..."  when SNK_DB_*.bin  — firmware Display Board
   → "LB IAP start ..."  when SNK_LB_*.bin  — firmware LED Board
   → "BTL IAP start ..." when SNK_MBTL_*.bin — firmware Bootloader
   → "DBL IAP start ..." when SNK_DBL_*.bin — firmware Dual?
```

### FORMATFLASH.json — USB Factory Reset

The string `"FORMATFLASH.json"` exists at `0x08003990`. Earlier analysis
found no direct code references in the main application (no LDR literal,
MOVW/MOVT, or LE32 pointers). However, the **MBTL bootloader** code
(also present in the same 1 MB flash) uses wildcard filename matching
on USB (`env_config*.json`, `SNK_MB_*.bin`, `btl_MB_*.bin`, etc.).
`FORMATFLASH.json` is matched by the same mechanism and triggers a
full flash erase.

Confirmed by cross-reference with the Brucke RM500 community
whose firmware shares identical strings and code paths.
The strings `"ready to format flash"` (`0x080039a4`) and
`"format flash\n"` (`0x080039bc`) are part of the same flow.

### What actually happens during format?

Format flash can be triggered:
1. **Via USB**: `FORMATFLASH.json` → MBTL bootloader erases entire flash
2. **Via command from app/display board**:
   - User sends command (via buttons or app over WiFi)
   - ESP32 → U16 (binary protocol)
   - U16 → U13 (JSON command)
   - U13 erases SPI flash (env) + EEPROM U22 (PIN)

The firmware contains battery pack definitions for BMS monitoring:

| String | Meaning |
|--------|---------|
| `5S1P_SUMSANG_20R` | 5S1P Samsung 20R (2000mAh) |
| `5S1P_SUMSANG_25R` | 5S1P Samsung 25R (2500mAh) |
| `5S1P_SUMSANG_29E` | 5S1P Samsung 29E (2900mAh) |
| `5S1P_SONY_VTC4` | 5S1P Sony VTC4 (2100mAh) |
| `5S2P_SONY_VTC4` | 5S2P Sony VTC4 (4200mAh) |
| `5S1P_EVE_2000` | 5S1P EVE 2000mAh |

BMS log format (from runtime): `5S1P_SONY_VTC4, id=xxxxxxxxx, voltage=20234, ocv=20234, percent=100, temp=16degree, charging times=113, discharge times=123, health=0`

The mower monitors only voltage and temperature — no software runtime limit
on battery capacity. Battery mods (larger cells or external batteries) work
without firmware modification.

## 6. EEPROM U22 — PIN / User Settings

### Hardware

| Parameter | Value |
|-----------|-------|
| IC | U22, SOIC-8 (labeled 24C02/04 in HARDWARE.md) |
| Interface | I2C2 (`0x40005800`) |
| Capacity | 256-512 bytes (24C02 = 2Kbit = 256B, 24C04 = 4Kbit = 512B) |
| Power | 3.3V (from mainboard) |
| I2C address | `0xD0` (8-bit) / `0x68` (7-bit) |

I2C2 functions in U13 code at: `0x08023550`, `0x080236C4`, `0x080237C0`.

### Stored data

From U13 firmware strings (module `service_user_set.c`):

| Category | Keys / Strings | Description |
|----------|----------------|-------------|
| **PIN** | `pwd`, `pwd_en`, `pwd_rst`, `usr_pwd_en` | Password, enable flag, reset flag |
| **Schedule** | `schedule`, `simple schedule` | Mowing schedule |
| **Calibration** | `battery calibrate`, `rtc calibration` | Battery and RTC calibration |
| **User** | `user_name`, `language`, `cut area` | Name, language, mowing area |
| **Counters** | `work length`, `power on minutes` | Hours of operation |

### PIN functions (from strings)

| Function | Code string |
|----------|-------------|
| Verify PIN | `compare pwd correct` / `compare pwd uncorrect=%d` |
| Set PIN | `set pwd success` / `set pwd failed` |
| Change PIN | `set pwd old failed, because input old password error` |
| Reset PIN | `reset pwd success` / `reset pwd failed` |
| Lock after failures | `compare pwd uncorrect=%d overtimes, reset and lock` |
| Enable/disable | `set user enable password failed=%d` |
| Load PIN | `load user password failed` |

### Rate limiting

The firmware has brute-force protection:
```
"compare pwd uncorrect=%d overtimes, reset and lock"
```
After exceeding the limit of failed attempts, the system **locks** ("reset and lock").
The attempt limit (parameter `%d`) is unknown but typically 3-5 attempts.

### PIN reset methods

PIN can be reset via:
1. **Old PIN** → `set pwd old failed, because input old password error` (requires old PIN)
2. **pwd_rst** → remote reset command (likely from the app)
3. **FORMATFLASH.json** — USB bootloader erases everything
4. **RAM read via SWD** — read PIN from `0x2000027C`

## 7. Env System (Key-Value Store)

U13 uses its own key-value storage system on SPI flash
(Winbond 25Q64JVSIQ, not to be confused with EEPROM U22).

| Operation | Function | Description |
|-----------|----------|-------------|
| Read | `FUN_0800aa6c(key, buf, size)` | Read parameter from env |
| Write | `FUN_0800ae28(key, buf, size)` | Write parameter to env |
| Delete | `FUN_0800a84c(key, 0, 1)` | Delete parameter |

**Stored parameters (firmware versions, config):**
- `MB_VER`, `mb_sv`, `MB_BVER`, `MB_SIZE`, `MB_BRF` — Main Board
- `BB_VER`, `bb_sv`, `BB_BVER`, `BB_SIZE`, `BB_BRF` — Boundary Board
- `DB_VER`, `db_sv`, `DB_BVER`, `DB_SIZE`, `DB_BRF` — Display Board
- `LB_VER`, `lb_sv`, `LB_BVER`, `LB_SIZE`, `LB_BRF` — LED Board
- `BTL_VER`, `BTL_BVER`, `BTL_SIZE`, `BTL_BRF` — Bootloader
- `pdt_ver`, `ota_date` — Product version, OTA date
- `cfgupdate`, `cfgstr` — Config update
- `cfg_rst` — Config reset (0xa5a5 two-step reset)
- `ult_cfg`, `led_cfg` — Sensor and LED config

## 8. Main Loop

`FUN_0800e800` — bitmask-driven dispatcher:

```
Init:
  FUN_08004238()      — init peripherals (GPIO, timers)
  FUN_08008948()      — init USB OTG
  FUN_0800cf04()      — init GPIO for buttons
  FUN_080024c4()      — init ADC (battery, sensors)
  FUN_08007b78()      — wait for U16 ready
  FUN_08007c68(0)     — init communication with U16
  FUN_080043ec()      — select USB mode (host/device)

Loop:
  ┌──────────────────────────────────────────────────────────┐
  │ bit 0  → FUN_08012930()    — USB protocol handler        │
  │ bit 1  → FUN_08003e0c()    — env_read.json load          │
  │ bit 2  → FUN_08002fc0()    — USB flash IAP               │
  │ bit 13 → FUN_0800303c()    — IAP/format flash            │
  │ bit 5  → FUN_08003cdc()    — save log to USB             │
  │ bit 6  → FUN_0800f54c()    — log/assert flush            │
  │ bit 14 → FUN_08001370()    — OTA over UART (from U16)    │
  │ always → FUN_0800bbf4()    — watchdog?                   │
  └──────────────────────────────────────────────────────────┘
```

## 9. OTA Update (two paths)

### OTA via USB (pendrive):
- User copies `SNK_MB_*.bin` etc. onto FAT32 pendrive
- U13 detects pendrive, finds files, programs its own flash
- Supports: MB, BB, DB, LB, BTL, DBL, DBS, DBH

### OTA via UART (from U16):
- U16 sends firmware to U13 over USART0
- 7 commands: GET OTA INFO, DOWNLOAD, SET OTA MODE, SET VER, RETURN, SET FIRMWARE NUMBER, SET BAUDRATE
- Variable baudrate: 230400 → 460800 → 921600
- XOR checksum on each packet
- ESP32 has no direct role in U13 OTA

## 10. Security Assessment — PIN Recovery

### Can the PIN be recovered?

| Method | Possible? | Notes |
|--------|-----------|-------|
| **KV cache in RAM (SWD)** | ✅ YES | PIN in RAM @ `0x2000027C` — just dump RAM |
| **USB pendrive (FORMATFLASH.json)** | ✅ YES | MBTL bootloader erases entire flash — factory reset |
| **Read EEPROM U22 (CH341A + SOIC clip)** | ✅ YES | Requires physical access to mainboard |
| **Reset via app (pwd_rst)** | ❓ UNKNOWN | If the app has a PIN reset command |
| **New firmware (IAP with SNK_*.bin)** | ❌ NO | IAP does not clear KV-store |
| **Brute-force via display** | ❌ NO | Rate limiting — lockout after X attempts |
| **From ESP32** | ❌ NO | ESP32 only stores WiFi password |
| **From U13 flash** | ❌ NO | PIN in RAM/EEPROM, not in U13 flash |

### Recommended PIN reset method (if forgotten):

1. **Factory reset via USB** — place empty `FORMATFLASH.json` on FAT32 pendrive (≤16 GB) into the internal USB port and power on. The MBTL bootloader will erase the entire flash.
2. **RAM read via SWD** — dump U13 RAM and read PIN from `0x2000027C` (our method).
3. **Physical EEPROM read** — CH341A + SOIC-8 clip on U22.

### Security notes

- PIN is cached in RAM at a fixed address (`0x2000027C`) — anyone with SWD access can read it
- MBTL bootloader accepts `FORMATFLASH.json` via USB — physical attack with a pendrive erases everything
- EEPROM (24C02) has no hardware security
- Rate limiting is only software (in U13) — can be bypassed
