# U16 Firmware Analysis — GD32F303CGT6 (Board MCU)

## Overview

U16 (GD32F303CGT6, ARM Cortex-M4 @ 72 MHz) is the **board/peripheral MCU** on the mainboard.
It handles all sensor acquisition, motor control logic, and UART communication with U13.

**420 functions** identified via Ghidra decompilation.

## Main Task: `comm_task` (FUN_08013680)

The main FreeRTOS task loop:

```
comm_task()
  loop:
    wait on queue (semaphore, INF timeout)
    if signal_flag:
      handle signal state machine
    
    if first_run:
      init factory driver with retry  (FUN_08017884)  // startup calibration
    else:
      run board process  (FUN_08014ac8)               // motor control logic
    
    deal_lift_sensor  (FUN_08013970)                  // lift sensor state machine
    dispatch_messages  (FUN_08019914)                  // send JSON messages
    
    delay 35 ticks (watchdog kick)
```

## Communication Protocol

### Physical Layer: UART (mport driver)

File: `driver_mboard_port_snk_v2.c`
Driver tag: `"mport drv"`

**Transmission (`send_string_dpport_snk_v2`, FUN_08019a18):**
```
send_string(data, length, param3, param4):
  if length >= 129:  // max 128 bytes
    log ERROR "send string length=%d too long"
    return error
  if length == 0:
    log WARN "@send len=0"
    return error

  // Wait for TX ready (semaphore with timeout)
  try 50000 iterations:  // ~busy-wait loop
    if semaphore_available:
      break
  if timeout:
    log ERROR "@wait send string overtime"
    return error

  // Send
  take_semaphore()
  clear_buffer(0x80)         // 128-byte buffer
  copy_data(data, length)
  set_length(length)
  enable_transmit()
  return success
```

**Reception:** Handled by `port_receive_callback` (callback registered during init).

### Data Format: JSON via cJSON

Messages are serialized using **cJSON** library. Key cJSON functions identified:
- `cJSON_Parse` / `cJSON_Print` (used at 0x08012fd4)
- `cJSON_AddItemToObject` (FUN_08012d78)
- `cJSON_AddItemToArray`
- `cJSON_GetObjectItem`
- `cJSON_CreateObject`, `cJSON_CreateNumber`, `cJSON_CreateString`

### Message Types

U16 sends these JSON messages to U13:

| Message | Trigger | Contents |
|---------|---------|----------|
| **border_message** | Flag at struct+0x12 == 0 | Border sensor data (left/right coil readings) |
| **version_message** | Flag at struct+0x14 == 0 | Hardware/software version |
| **seach_border_message** | Flag at struct+0x18 == 0 | Search border mode status (note: typo "seach" in original) |
| **follow_border_message** | Flag at struct+0x19 == 0 | Follow border mode status |

The message dispatcher (FUN_08019914) reads these flags from the board sensor data structure and builds JSON using cJSON operations.

### Incoming Commands

U13 sends JSON commands to U16. The JSON parser (`get_process_comm`, FUN_080180a0 / FUN_08017f92) handles:
- Objects `{}` — recursive key-value parsing
- Arrays `[]` — recursive element parsing
- Strings `""` — with escape sequence support (\\, \", \n, \r, \t, \f, \b, \uXXXX)
- Numbers — integer, float with sign, decimal point, exponent
- Booleans — `true` / `false` (memcmp)
- Null — `null` literal

Parser returns typed tokens:
| Type code | Meaning |
|-----------|---------|
| 0x2 | boolean (true) |
| 0x4 | null |
| 0x8 | number (double) |
| 0x10 | string |
| 0x20 | array |
| 0x40 | object |

Error handling: "error cmd =%d", "message point is null", "get message cmd is null", "get message string is null"

## Sensors

### Lift Sensors (deal_lift_sensor, FUN_08013970)

Two hall-effect sensors detect when the mower is lifted.

State machine (state stored at struct offset 0x0e):

| State | Meaning | Detection |
|-------|---------|-----------|
| 0 | `lift none` | Neither sensor triggered |
| 1 | `both lifted` | Both sensors triggered |
| 2 | `left lifted` | Left sensor triggered |
| 3 | `right lifted` | Right sensor triggered |

Debounce: 10 consecutive samples required to confirm state change.
Counter variables at struct offsets 0x0f-0x11 track sample counts for each state.

### Border Wire Sensors (bdsensor, SPI)

File: `driver_board_sensor/driver_bdsensor.c`
Two coils under the chassis sense the electromagnetic field from the boundary wire.

SPI protocol with magic frame headers:
| Pattern | Address | Purpose |
|---------|---------|---------|
| `"H@h80pGx"` | 0x08016c8d | Left sensor data frame start |
| `"H@hX0pGx"` | 0x08016e49 | Right sensor data frame start |
| `"H@h 0pGx"` | 0x08016e6d | Another frame header |
| `"IH\`pG"` | 0x08019db5 | SPI command |

Sensor acquisition (`bdsensor_init`):
1. Enable GPIO clocks (pins on ports 5, 6, 7, 0xA)
2. Configure SPI peripheral (prescaler 0x200)
3. Configure PWM timers (for sensor excitation)
4. Read base voltage 5 times via SPI, compute average
5. Store as baseline (`DAT_08012c88`)
6. Log `"base voltage=%d"`

Data collection (`start_collect_data`):
- Waits for 1000+ samples in buffer
- Uses dual 16-byte buffers for ADC data
- Timer period: 18868 (at 0x92 in struct)
- Sample buffer size: 800

### Voltage Monitoring

Base voltage measured via SPI ADC (5-sample average).
Used for battery monitoring: `"base voltage=%d error, set default 2048"`

## Motor Control (FUN_08014ac8)

**YES, U16 controls motors.** It reads border sensor status and makes motor decisions.

### PWM Timer Configuration (FUN_08019bf4)

Two PWM timer channels configured:

| Channel | Timer Base | Period (ARR) | Notes |
|---------|-----------|--------------|-------|
| Channel 1 | `DAT_08019cf0` | 0x60000 (393216) | Prescaler = 1, Mode: output compare |
| Channel 2 | `DAT_08019cf4` | 0xe0000 (917504) | Prescaler = 1, Mode: output compare |

Configuration:
1. Set alternate function for timer output pins (via FUN_08012aa2)
2. Configure output compare mode
3. Enable preload
4. Set polarity for pins (ch1: pins 5/7, ch2: pins 9/7)
5. Enable output
6. Enable brake/break
7. Generate update event
8. Enable timer + set bit 8 in control register (main output enable)

### Border Following Logic (FUN_08014ac8)

```
board_process():
  left_status = sensor_vtable[+8]().status    // left coil
  right_status = sensor_vtable[+0xc]().status  // right coil

  // Status values:
  //   0x00 = normal / no signal
  //   0x01 = signal in range (inside border)
  //   0x02 = over range
  //   0x03 = over limit (strong signal)

  if left == 0x03 OR right == 0x03:
    factory_vtable[+12]()      // STOP motors (over-limit = at wire)
  
  else if left == 0x01 AND right == 0x00 AND left_raw > 29000 AND right_raw < -29000:
    factory_vtable[+12]()      // STOP (left inside, right outside, strong gradient)
  
  else:
    counter = struct[0x17]     // border follow debounce counter
    if counter < 20:
      factory_vtable[+8]()     // MOVE FORWARD (inside area)
    else if counter < 40:
      factory_vtable[+12]()    // TURN/STOP (approaching boundary)
    else if counter < 80:
      factory_vtable[+8]()     // MOVE FORWARD (turning away from boundary)
    else:
      counter = 0              // reset cycle
```

**Factory driver vtable:**
- vtable[+8] = move forward / drive motors
- vtable[+12] = stop / turn

This implements a classic **border following algorithm**: the mower follows the perimeter wire by detecting the magnetic field from each coil and executing a pattern of forward/turn cycles when approaching the boundary.
